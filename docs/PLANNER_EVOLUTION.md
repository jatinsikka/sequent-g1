# The brain: what we trained, what we kept, what we dropped

*Research log, updated 2026-06-30. Every number here was measured by running the code this session (or
cross-checked on a second machine). Where we have no measurement, we say so — we do not fabricate.*

## TL;DR

The brain does two jobs: **retrieve** the right SOP for a plain-English incident, then **plan** it into a
chain of robot skills. We trained our own model for each. One worked; one didn't. The honest decision was
never "custom vs frontier" — it was task by task.

| Component | What we trained | Result (measured) | Decision |
|---|---|---|---|
| **Retriever** | fine-tuned MiniLM sentence bi-encoder (MNR loss) | **R@1 0.57** vs TF-IDF 0.30 on 269 held-out incidents | **kept** — it's the RAG layer |
| **Planner** | Flan-T5 seq2seq + LoRA | **0/20 valid plans** (re-run twice, independently) | **dropped** → frontier LLM |

**The moat is neither of these.** Planning is something a frontier LLM does better than anything we could
train from scratch, so we use one. Our defensible contribution is the layer no API gives you: **every planned
skill is verified against measured physics before it counts.** The model proposes; the physics disposes.

## The retriever — a real custom-trained win (kept)

We fine-tuned `all-MiniLM-L6-v2` (a 22M-param BERT-architecture bi-encoder) with a multiple-negatives ranking
loss to map an operator's plain-English incident to the correct SOP. Measured **live** (eval-only, loading the
saved model) on **269 held-out** test incidents over a 1,012-SOP corpus:

| method | R@1 | R@5 | MRR |
|---|---|---|---|
| TF-IDF (lexical) | 0.30 | 0.54 | 0.41 |
| **fine-tuned bi-encoder** | **0.57** | **0.87** | **0.69** |
| bi-encoder + cross-encoder rerank | 0.44 | 0.83 | 0.60 |

Numbers come from real code (`encode → cosine → argsort → rank metrics`), not constants — confirmed because the
reranker row is genuinely *worse* (0.44), which a hardcoded table would never show.

**Honest caveats (do not oversell):** the test set is **held out from training but same-distribution synthetic**
(incidents generated by the same model/prompt as the corpus). There is **no out-of-distribution test** and **no
gold reference plans**. So the defensible claim is exactly: *"a fine-tuned bi-encoder beats lexical and naive-neural
baselines on a hard, held-out, same-distribution retrieval benchmark."* Not "robust," not "generalizes" — unmeasured.

## The planner — a custom model that failed (dropped)

We also trained our own planner: a `Flan-T5-base` seq2seq with LoRA, to emit the skill chain as JSON. It does not
work. Run on 20 test incidents — **twice, independently (my machine and a second one)** — it produces **0/20 valid
JSON**: it echoes the prompt's literal placeholder (`"sop title>"`) and emits malformed, unparseable output. There
was no small fix; the model never learned the structure.

A prior writeup reported **F1 0.78 / 95% exec / 98% valid-JSON** for this planner. Those were **hardcoded
placeholders** in the eval script (`{"r":32,"f1":0.78}` with a `# Simulated` comment; the heuristic F1 was
`f1_scores.append(0.5) # rough estimate`). They were never measured, and the model's actual 0% output contradicts
them. They are not used anywhere now.

## Why a frontier model for planning, and where our originality goes

Planning rewarded the opposite call from retrieval. A frontier LLM (Claude / Gemini) reasons over the top-k
retrieved SOPs, picks the one that fits, and writes valid, sensible skill chains out of the box — inferring the
navigation SOPs leave implicit — where our from-scratch planner produced nothing usable. Spending months matching
a frontier model on language→plan would be wasted effort.

We **do not** claim a planner-*accuracy* number: this corpus has no gold reference plans, so any F1 would be
invented. We won't. What we can say is exact and measured: the trained LoRA planner = 0% valid; the frontier LLM
produces valid plans and is the production planner.

So the planner is a frontier model, the retriever is our own fine-tuned bi-encoder, and the research contribution
is the verification layer — runtime physics-feasibility checking of each planned skill ("a humanoid that knows
when it can't"). That is the part that is novel and defensible; the planner choice is a pragmatic engineering
decision in service of it.
